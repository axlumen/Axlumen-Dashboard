# Analysis of table-github-capability-router Project

## Project Overview

The `table-github-capability-router` is a Markdown-first workflow system for:
1. **GitHub project intake** - Evaluating and archiving GitHub projects
2. **Capability cold storage** - Managing skills, plugins, MCP servers, scripts, and CLIs
3. **Agent routing** - Creating a layered routing system to control what the agent sees

## Key Design Patterns Identified

### 1. Layered Routing Architecture
- **L0**: Always-visible rules files
- **L1**: Thin router (task categories, truncation-safe one-liners)
- **L2**: Registry rows (triggers + do-not-use conditions)
- **L3**: Capability card (full detail of one capability)
- **L4**: Capability itself (loaded only when enabled)

### 2. Truncation-Safe Writing Rules
- First ~100 characters express purpose independently
- Trigger scenarios and do-not-use conditions within first ~200 characters
- "Do not use for..." never at the end
- No reliance on question marks or contrastive words
- One entry serves exactly one task type

### 3. Manager-Type Capability Governance
Identifies capabilities that try to take over routing:
- Startup-wide triggers ("Use when starting any conversation")
- Default-flow rewrites
- Self-reinforcement patterns
- Light surface, heavy effect
- Methodology takeover
- SessionStart hook injections

### 4. Cold Storage State Model
- **active**: Useful and safe by default
- **cold**: Useful but requires deliberate activation
- **disabled**: Installed but should not be routed
- **reference**: Method knowledge, not executable tool
- **retired**: No longer used

### 5. Evidence-Based Decision Making
Evidence levels from author claims to real workflow validation:
- Author claim → Online check → Static check → T0 → T1 → T2 → T3

### 6. Knowledge Distillation Pattern
Extracting reusable methods from projects and placing them in appropriate locations:
- Workflow pages (repeatable processes)
- Concept pages (general ideas)
- Project pages (project-specific lessons)
- Project cards (source evidence)
- Reference manifests (agent-readable metadata)

## Target Plugin: GullDock (Axlumen Dashboard)

**Platform**: Obsidian plugin
**Current State**: Agent-First Vault Intelligence Dashboard with:
- Agent system (AgentRunner + Claude Code CLI)
- Automation engine with trigger system
- Inbox routing (AI-prompt-based)

- No formal capability registry

## Risk Mitigation

1. **Backward Compatibility**: Keep existing workflow system working during migration
2. **Performance**: Lazy-load capabilities, cache routing decisions
3. **Complexity**: Start with minimal viable registry, iterate based on usage
4. **User Experience**: Maintain existing UI patterns, add new features incrementally
