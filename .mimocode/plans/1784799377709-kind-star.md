# TGCR × GullDock 代码审查报告

## 总览

6 个 Feature 共新增 13 个文件，修改 4 个现有文件。以下是逐模块审查发现的问题和改进建议。

---

## Feature 1: 薄路由层 (WorkflowRegistry + RoutingEngine)

### 问题

**P1 — `no-internet` 禁用条件逻辑错误** (`WorkflowRegistry.ts:123`)
```ts
if (c === 'no-internet' && context.inboxCount < 0) return true;
```
`inboxCount` 不可能 < 0，应该用其他方式检测网络状态，或者移除这个条件。

**P2 — token 匹配存在重复计分** (`WorkflowRegistry.ts:86-94`)
```ts
for (const pt of patternTokens) {
  for (const it of inputTokens) {
    if (it === pt || it.includes(pt) || pt.includes(it)) {
      score += 3;
    }
  }
}
```
`it.includes(pt)` 和 `pt.includes(it)` 是双向包含检查，会导致 "inbox" 和 "in" 互相匹配产生重复加分。应该只保留一个方向。

**P3 — RoutingEngine 缺少缓存失效机制** (`RoutingEngine.ts:56`)
```ts
if (this.registry.getAll().length === 0) {
  await this.syncFromWorkflows();
}
```
只在空时同步一次，之后 workflow 模板变化不会自动更新 registry。需要增加缓存失效策略。

**P4 — RoutingEngine.route() 未暴露给外部调用**
`route()` 和 `routeWithCandidates()` 方法已实现，但没有被任何地方调用。AgentRunner 的 `run()` 方法仍然直接传 workflowId，没有经过路由层。

### 建议

1. 移除 `no-internet` 条件或改用 Obsidian 的 `navigator.onLine`
2. token 匹配改为单向：只检查 `inputTokens` 是否包含 `patternTokens` 的元素
3. 增加 `syncVersion` 计数器，workflow 扫描时递增，route() 检查版本号
4. 在 AgentRunner.run() 中增加可选的路由步骤，或提供 `routeAndRun(userInput)` 方法

## Feature 2: 能力冷库 (CapabilityManifest)

### 问题

**P1 — `recordFailure` 的健康升级逻辑有缺陷** (`CapabilityManifest.ts:112`)
```ts
if (entry.failureCount > 3 && entry.runCount < 10) {
  entry.health = 'broken';
} else if (entry.failureCount > 1) {
  entry.health = 'degraded';
}
```
`failureCount` 是累积值，不是连续失败次数。一个运行了 100 次、失败 4 次的健康 workflow 会被标记为 `broken`。应该追踪连续失败次数或使用失败率。

**P2 — `syncFromWorkflows` 无条件覆盖元数据** (`CapabilityManifest.ts:179-186`)
已存在的条目会被覆盖 `name`、`description`、`purpose` 等字段，但不会更新 `source`。如果一个 workflow 的 source 从 `workflow` 变为 `builtin`，不会被同步。

**P3 — `saveCallback` 在 `load()` 时不触发**
`load()` 只填充数据不调用 `saveCallback()`，这是正确的。但 `syncFromWorkflows()` 和 `syncFromIntegrations()` 都会调用 `saveCallback()`，即使没有实际变化也会触发写入。

**P4 — `inferSlot` 是硬编码的关键词匹配** (`CapabilityManifest.ts:240-249`)
无法正确推断用户自定义 workflow 的 slot。应该允许在 frontmatter 中显式声明 slot。

### 建议

1. 改为追踪 `consecutiveFailures` 或使用 `failureCount / runCount` 比率
2. sync 时检查 source 是否变化，变化时更新
3. sync 前比较是否有实际变化，无变化则跳过 saveCallback
4. 支持 frontmatter 中的 `slot` 字段

## Feature 3: 截断安全写作 (TruncationGuard)

### 问题

**P1 — `disableWhen` 检查对空值不健壮** (`TruncationGuard.ts:69`)
```ts
const disableTerms = template.disableWhen.toLowerCase().split(',').map(s => s.trim());
```
如果 `disableWhen` 是空字符串 `""`，`split(',')` 返回 `[""]`，`trim()` 后是 `""`，`lastLines.includes("")` 永远为 true。会导致所有模板都报错。

**P2 — VERB_PATTERN 只匹配中文动词** (`TruncationGuard.ts:27`)
英文 prompt 的前 3 行不会被检查动词。对于英文 workflow 模板，这个规则完全失效。

**P3 — `truncatePurpose` 的句号截断逻辑有边界问题** (`TruncationGuard.ts:146-151`)
```ts
const lastPeriod = Math.max(
  truncated.lastIndexOf('。'),
  truncated.lastIndexOf('.'),
  ...
);
```
`Math.max` 在所有 `lastIndexOf` 返回 -1 时返回 -1，`-1 > maxLen * 0.6` 为 false，会走 `truncated + '...'` 分支。这是正确的，但如果 `maxLen` 是 0 会有问题。

### 建议

1. 在 split 后 filter 掉空字符串：`disableTerms.filter(t => t.length > 0)`
2. 增加英文动词模式：`/^(?:analyze|scan|check|generate|create|evaluate|process|run|route)/mi`
3. `truncatePurpose` 增加 `maxLen > 0` 的前置检查

## Feature 5: 总管型审计 (ManagerAudit)

### 问题

**P1 — `checkOverride` 误报率高** (`ManagerAudit.ts:106`)
```ts
return /replace|override|instead\s+of|bypass|skip|不使用|替代|覆盖默认|绕过|取代/.test(text);
```
这些词在正常 workflow 描述中也很常见（如 "替代手动操作"、"跳过重复步骤"）。会导致大量误报。

**P2 — `checkMethodologyTakeover` 的 step 计数不区分层级** (`ManagerAudit.ts:121`)
```ts
const customSteps = (promptLower.match(/step\s+\d+|第[一二三四五六七八九十\d]+步/g) || []).length;
```
简单的 step 计数不能区分 "3 个子步骤" 和 "8 个顶层步骤"。一个正常的 5 步 workflow 会被误判为 methodology-takeover。

**P3 — `checkStartupHook` 只检查文本匹配，不检查实际 Hook 配置**
TGCR 原文说 "SessionStart Hook 注入被按定义视为总管型"，但当前实现只做文本匹配，不检查实际的 Hook 注册。

### 建议

1. `checkOverride` 改为只匹配明确的替换意图短语（如 "替代标准流程"、"绕过正常流程"），而不是单个词
2. step 计数改为只计算顶层步骤（不缩进的 step）
3. 增加对 Obsidian Plugin Hook 的检测（如果可行）

## Feature 6: 隐私脱敏 (PrivacySanitizer)

### 问题

**P1 — `scan()` 中 `rule.pattern.lastIndex = 0` 放在循环内有性能问题** (`PrivacySanitizer.ts:216`)
对每一行都重置 `lastIndex`，但 global regex 在 `exec()` 时会自动推进 `lastIndex`。应该在每行开始时重置一次，而不是在循环内。

**P2 — `sanitize()` 中的替换可能遗漏** (`PrivacySanitizer.ts:250-253`)
```ts
for (const finding of findings) {
  rule.pattern.lastIndex = 0;
  sanitizedContent = sanitizedContent.replace(rule.pattern, rule.replacement);
}
```
`String.replace()` 只替换第一个匹配。如果一行有多个 API Key，只有第一个会被替换。应该用 `replaceAll` 或循环替换。

**P3 — `loadRules` 的 flags 推断不准确** (`PrivacySanitizer.ts:290`)
```ts
pattern: new RegExp(r.pattern, r.pattern.endsWith('i') ? 'gi' : 'g'),
```
只检查 pattern 是否以 'i' 结尾，但如果原始 pattern 是 `/pattern/gmi`，序列化后 source 是 `pattern`，flags 信息丢失。

**P4 — OpenAI key 正则不匹配新版 key 格式** (`PrivacySanitizer.ts:124`)
```ts
pattern: /sk-[a-zA-Z0-9]{48}/g,
```
新版 OpenAI key 格式是 `sk-proj-...`（更长），当前正则无法匹配。

### 建议

1. `lastIndex = 0` 移到 for 循环外（每条规则开始时重置一次）
2. 用 `replace(new RegExp(rule.pattern.source, 'g'), rule.replacement)` 替换
3. 序列化时同时保存 flags：`pattern: r.pattern.source, flags: r.pattern.flags`
4. 更新 OpenAI key 正则：`/sk-(?:proj-)?[a-zA-Z0-9\-_]{20,}/g`

## Feature 4: GitHub 入库 (GitHubIntakeService)

### 问题

**P1 — `fetchViaGhCli` 永远返回失败** (`GitHubIntakeService.ts:211-213`)
```ts
private async fetchViaGhCli(...): Promise<ServiceResult<GitHubProjectInfo>> {
  return err('GH_CLI_UNAVAILABLE', 'gh CLI 不可用', true);
}
```
注释说 "gh CLI 在 Obsidian 沙箱中不可用"，但 AgentRunner 已经有 `getNodeRequire('child_process')` 的沙箱检测逻辑。应该复用同样的逻辑尝试 gh CLI。

**P2 — `fetchViaApi` 没有处理 rate limiting** (`GitHubIntakeService.ts:219`)
GitHub API 未认证请求限制为 60 次/小时。没有处理 429 响应。

**P3 — `evaluateVeto` 只检查 description 中的 "archived"** (`GitHubIntakeService.ts:85`)
应该检查 GitHub API 返回的 `archived` 字段（布尔值），而不是 description 文本匹配。

### 建议

1. 复用 AgentRunner 的 `getNodeRequire` 逻辑尝试 `gh api` 命令
2. 增加 429 响应处理和 rate limit 检查
3. 使用 `data.archived` 字段而非 description 文本匹配

## 集成问题 (main.ts)

**P1 — `saveSettings` 每次调用都同步 TGCR 数据** (`main.ts`)
```ts
async saveSettings() {
  this.settings.capabilityManifest = this.capabilityManifest.serialize();
  this.settings.privacySanitizeRules = this.privacySanitizer.serializeRules() as ...;
  ...
}
```
即使 TGCR 数据没有变化，每次保存设置都会序列化并写入。应该只在 TGCR 数据变化时才同步。

**P2 — `lint-workflow-templates` 命令的 Notice 显示**
```ts
new Notice(`Workflow Lint: ${results.length - errors - warnings} 通过, ${warnings} 警告, ${errors} 错误`);
```
没有展示具体的 lint 结果。用户看不到哪个模板有什么问题。应该打开一个面板或 modal。

**P3 — 缺少 `github-intake` 命令的参数传递**
```ts
this.addCommand({
  id: 'github-project-intake',
  name: 'Agent: GitHub Project Intake',
  callback: () => this.runWorkflow('github-intake'),
});
```
没有提供输入 GitHub URL 的方式。用户需要手动编辑 workflow 模板的 `{{input}}` 参数。
